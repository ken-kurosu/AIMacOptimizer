# AI Mac Optimizer — Pro 継続価値戦略レポート

作成日: 2026-07-08
対象: Free/Pro 課金設計の深掘りと「使い続けるほど効く」継続価値（recurring value）の設計

---

## 0. エグゼクティブサマリー

現在の Pro（¥480/月 / ¥4,980 買い切り）の価値は「ストレージ削除の手間削減」と「スケジュール自動化」に集約されており、**一度きりの利便性**であって**継続する価値**ではない。これがサブスク転換の弱さと買い切りへの流出の根本原因。

本レポートの結論は 3 行で言える:

1. **Pro の主語を「削除できる」から「あなたの Mac を知っている」へ移す。** HealthHistory / PatternLearner が蓄積するデータを原資に、「定期最適化レポート」を Pro の顔にする。データは日々溜まるので、価値は使うほど増え、解約するほど失うものになる。
2. **Free は今のまま実用的に保つ。** 壁は「機能の有無」ではなく「時間軸の深さ（履歴・トレンド・予測）」と「自動化の度合い」に置く。これは「正直さ」ブランドと矛盾しない — 正直な計測は無料、正直な計測の**蓄積と解釈**が有料。
3. **買い切りは廃止せず「Pro 機能のスナップショット」として据え置き、月額を「育ち続けるプラン」として明確に差別化する。** レポートの高度化・新検出ルール・複数 Mac 等、継続的に増える価値は月額側に寄せる。

## 1. 継続価値（recurring value）の設計原則

### 1.1 なぜ今の Pro はサブスクとして弱いのか

| Pro 機能（現状） | 価値の性質 | サブスク適性 |
|---|---|---|
| ストレージのファイル削除 | 手間削減（Free のスキャン結果を手動で消せば同じ） | ✕ 一度きり。しかも Free が場所を教えてしまう |
| スケジュール自動最適化 | 自動化 | △ 継続的だが「効いている実感」が見えない |
| 優先サポート | 保険 | △ 使わない月は価値ゼロ |

問題の構造: **Free のスキャンが「診断書」を無料で渡し、Pro は「手術」だけ売っている**。診断書があれば手術は Finder で自分でできる。だから転換しない。

### 1.2 サブスクを正当化する 3 つの価値類型

サブスクが成立するのは、価値が以下のいずれかに該当する場合:

1. **蓄積価値（history）** — 使った期間に比例してデータ・文脈が溜まり、他では再現できなくなる。解約 = 蓄積の喪失。
   - 本アプリの原資: `HealthHistory`（10分間隔の時系列）、`PatternLearner`（アプリ別利用プロファイル）
2. **自動化価値（automation）** — ユーザーが寝ている間も働く。「月額 ¥480 で毎日番人が立っている」という納得感。
   - 原資: `ScheduleManager`、`DiskGuard`、`NotificationService`
3. **助言価値（personalized advice）** — 汎用 Tips ではなく「あなたの Mac の、あなたの使い方」に基づく指摘。データが溜まるほど精度が上がる。
   - 原資: `SmartAdvisor` + `PatternLearner` + `HealthHistory` の掛け合わせ

**この 3 つを 1 本に束ねる器が「定期最適化レポート」**である。レポートは (1) 履歴がないと書けず、(2) 自動で届き、(3) パーソナライズされている。つまり recurring value の 3 類型を全部満たす、Pro の理想的なフラッグシップ機能。

### 1.3 「実測で正直」ブランドとの整合

クリーナー系が「32GB 解放できます！」と盛るのに対し、本アプリは実測値だけを出す。レポートも同じ原則で作る:

- **予測を約束しない。実績を報告する。**「今月あなたの Mac で実際に解放したメモリ: 合計 41.2GB（14 回の最適化）」
- **問題がなければ「問題なし」と書く。**「今週は健康です。やることはありません」と言えるレポートは、クリーナー系には絶対に作れない差別化になる。
- これにより、レポート自体が「正直さ」ブランドの毎週の証明書になる。

## 2. 中核機能:「定期最適化レポート / 通知」詳細仕様

### 2.1 コンセプト

> **「あなたの Mac の週次健康診断」** — 何が容量を食ったか、快適さは維持できているか、次に何をすべきかを、実測データだけで毎週報告する。

### 2.2 集計する内容（データソースと算出方法）

すべて既存の内部機能から算出可能。新規のデータ収集基盤はほぼ不要。

| セクション | 内容 | データソース | 算出方法 |
|---|---|---|---|
| **① 容量を食った犯人 Top 5** | 期間中に増加したディレクトリ/キャッシュのランキング | `StorageAnalyzer.scan()` の結果を週次で永続化した差分 | 前回スナップショットとの差分（例: 「DerivedData +8.4GB」「Adobe Media Cache +3.1GB」）。**新規実装: スキャン結果のスナップショット保存**（後述 5.2） |
| **② 空き容量トレンド** | 空き容量の推移グラフ＋「このペースだと約 N 週間で 10% を切る」 | `HealthHistory.diskFreePercent`（既存） | 直近 7〜14 日の線形回帰。予測は「このペースなら」という条件付き表現で正直に |
| **③ メモリ/Swap 傾向** | Swap 常用の有無、メモリ圧迫の時間帯 | `HealthHistory.swapMB` / `memUsedPercent`（既存） | 週平均・ピーク・「Swap が 2GB を超えていた時間: 週 14 時間」など。時間帯分析は `PatternLearner` の hour 別プロファイルと突合 |
| **④ 劣化兆候の検出** | 「先週より重くなっている」シグナル | `HealthHistory.loadAvg1` + `swapMB` の週次比較 | 週平均の前週比。例: 「平均 Swap が先週比 +40%。Chrome のタブ数増が主因の可能性（タブ分析: 平均 34 → 51）」 |
| **⑤ 最適化の実績** | 実際に解放したメモリ/容量の累計 | `MemoryOptimizer` / 削除実行のログ（**新規: 実行ログの記録**） | 「今週の実測解放: メモリ 12.3GB（4 回）、ディスク 9.8GB」— 実測値のみ。推定は出さない |
| **⑥ 今週のアドバイス（1〜3 個）** | パーソナライズされた次のアクション | `SmartAdvisor` + `PatternLearner` | 後述 2.5 |

### 2.3 頻度・タイミング

| 種別 | 頻度 | タイミング | 根拠 |
|---|---|---|---|
| **週次レポート** | 週 1 | 月曜 9:00（変更可） | ビジネスマンの「週初めに整える」習慣に接続。金曜夕方案もあるが、週末に Mac を触らない層が読まない |
| **月次サマリー** | 月 1 | 月初第 1 営業日 | 「今月の実測解放 合計」「容量の増減収支」を数字で総括。**課金更新日の直前に価値を再認識させる**効果（更新日 3 日前に寄せるのも有効） |
| **臨時アラート（既存拡張）** | 随時 | 閾値超過時 | `NotificationService` の既存閾値通知に「文脈」を追加（後述 2.6） |

頻度はユーザー設定で「週次/隔週/月次のみ/オフ」を選べるようにする。**通知疲れは解約の最大要因**なので、デフォルトは控えめ（週次 1 通）に。

### 2.4 チャネルと見せ方

**優先順位: ① アプリ内レポート画面 → ② macOS 通知（サマリー） → ③ メール（任意・後回し）**

1. **アプリ内「レポート」タブ（本体）**
   - `HealthTrendView` の隣に「Weekly Report」ビューを追加。過去レポートのアーカイブも閲覧可（Pro は全期間、Free は最新 1 件のみ — 履歴の壁）。
   - 構成: 冒頭に健康スコア 1 行（「今週の総合: 良好 🟢」）→ ①〜⑥ をカード形式で。グラフは HealthTrendView の描画コンポーネントを再利用。
2. **macOS 通知（入口）**
   - 通知はレポートの「見出し」だけ: 「📊 週次レポート: DerivedData が +8.4GB。空き容量は 3 週間で 10% を切るペースです」→ クリックでアプリ内レポートを開く。
   - 通知文にはその週**最も重要な 1 事実**だけを載せる。全部載せると読まれない。
3. **メール（フェーズ 2 以降・任意）**
   - サーバーインフラとメアド収集が必要になるためコスト高。まずはローカル完結（通知＋アプリ内）で出す。オンデバイス主義とも整合する。メールは「複数 Mac 対応」を作るタイミングでまとめて検討。

### 2.5 パーソナライズの具体（PatternLearner / HealthHistory の活用）

汎用 Tips とパーソナライズの違いを明確にする。以下はすべて既存データで生成可能な文例:

- **アプリ別の癖の指摘**（PatternLearner の `AppUsageProfile`）:
  - 「Slack は起動から時間が経つと平均 1.8GB まで膨らみます（あなたの実測）。週 1 回の再起動で平均 1.2GB 軽くなっています」
  - `timesOptimized` を使い「あなたが過去に承諾した最適化」を学習 → 承諾率の高い提案を優先表示、無視され続けた提案は出さない（通知疲れ対策）
- **時間帯パターン**（PatternLearner の hour 別記録 × HealthHistory）:
  - 「毎週火・木の 14〜16 時にメモリ圧迫が集中しています（会議＋Chrome の時間帯？）。この時間帯の直前に自動最適化をスケジュールしますか？」→ **そのままスケジュール設定への導線（Pro 機能への送客）**
- **開発者向けの文脈検出**（StorageAnalyzer の検出項目から職種を推定）:
  - DerivedData が検出される → 「Xcode ユーザー向け: ビルドのたびに DerivedData が育ちます。あなたの環境では週平均 +5.2GB。月 1 の自動クリーンが有効です」
  - Adobe Media Cache → クリエイター向け文言、Cursor/VS Code → 開発者向け文言に切り替え
- **劣化の因果仮説の提示（断定しない）**:
  - 「Swap 増加と Chrome タブ数増加が同じ週に起きています。因果は断定できませんが、タブ整理を試す価値があります」— 正直さブランドに沿った控えめな表現をテンプレートで統制する

### 2.6 Free 版（軽量）と Pro 版（フル）の線引き

**原則: Free は「今の事実」、Pro は「時間軸と自動化」。**

| 要素 | Free | Pro |
|---|---|---|
| 週次レポート通知 | ○ 届く（重要 1 事実＋健康スコアのみ） | ○ フル |
| ① 容量犯人 Top | Top 1 のみ（「一番増えたのは DerivedData +8.4GB」） | Top 5 ＋週次推移＋ワンクリック回収ボタン |
| ② 空き容量トレンド | 直近 24h のみ | 全期間グラフ＋枯渇ペース予測 |
| ③ Swap/メモリ傾向 | 現在値のみ | 週次トレンド・時間帯ヒートマップ |
| ④ 劣化兆候 | — | ○（前週比分析） |
| ⑤ 実測解放の累計 | 当日分のみ | 週次/月次/全期間累計（「累計 214GB 解放」はロイヤルティの数字にもなる） |
| ⑥ パーソナライズ助言 | 汎用アドバイス 1 件 | パーソナライズ 3 件＋実行導線 |
| レポートアーカイブ | 最新 1 件 | 無制限 |

**この設計の狙い**: Free ユーザーにも毎週レポート通知が届く。つまり**毎週 1 回、Pro のフルレポートの「予告編」を見る**ことになる。「一番増えたのは DerivedData +8.4GB」まで見せて、「他の 4 つと推移を見る → Pro」で止める。Free の実用性（今の事実は全部わかる）は削っていないが、「先週と比べてどうか」「今後どうなるか」「どうすればいいか」は Pro。これは情報の出し惜しみではなく、**蓄積・解釈という追加労働への対価**なので、正直さブランドと両立する。

### 2.7 レポート生成の実装方式

- 完全ローカル生成。週次タイマー（`ScheduleManager` のタイマー機構を流用）で `ReportGenerator`（新規サービス）を起動。
- 生成物は JSON（構造化データ）＋描画は SwiftUI。`~/Library/Application Support/AIMacOptimizer/reports/2026-W28.json` のように保存。
- AI 相談（LocalAdvisor / AppleIntelligence）との連携: レポート画面に「このレポートについて AI に相談」ボタン → レポート JSON をコンテキストとしてチャットに注入。**既存のローカル AI 相談が「文脈を持った相談」に進化**し、Free の AI 相談（汎用）との差も自然に生まれる。

## 3. その他の「使い続けるほど効く」レバー — 優先度マトリクス

各案を **転換インパクト（Free→Pro 転換への寄与）× 実装コスト** で評価。◎>○>△。

| # | 施策 | 内容 | 転換インパクト | 実装コスト | 優先度 |
|---|---|---|---|---|---|
| L1 | **履歴保持期間の差** | HealthHistory / レポートアーカイブ / 解放実績を Free 24h・Pro 無制限に。HealthTrendView のグラフに「ここから先は Pro」のロック表示 | ◎（蓄積価値の壁。最も王道） | **低**（既存 `maxAgeSec` の分岐＋UI ロックのみ） | **P0** |
| L2 | **定期レポート（第 2 章）** | Pro の顔となる中核機能 | ◎ | 中（2〜3 週間規模） | **P0** |
| L3 | **隠れた大物のワンクリック回収の拡充** | 現状の StorageAnalyzer は Xcode DerivedData / Adobe / Cursor / Chrome 系を検出済み。**未対応の ollama モデル（`~/.ollama/models`、数十 GB 級）、iOS シミュレータ（`~/Library/Developer/CoreSimulator`、`xcrun simctl delete unavailable` で安全削除可）、Xcode 旧 iOS Device Support、Docker.raw、node_modules（プロジェクト横断）、Homebrew キャッシュ** を追加 | ○〜◎（ターゲット層に「数十 GB 級の一撃」。レポートの①に載る弾でもある） | 低（`cachePaths` 配列への追加＋safety 分類。シミュレータのみ simctl 連携） | **P0** |
| L4 | **DiskGuard の主役化（自動ディスクガード）** | 現在裏方の DiskGuard を Pro の看板に: 「空きが 10% を切ったら、安全なキャッシュだけ自動回収して通知」。実行後レポート「DiskGuard が 4.2GB を自動回収しました（内訳）」 | ◎（「寝ている間も守る」= 月額の納得感が最も直感的） | 低〜中（DiskGuard 本体は実装済み。Pro ゲート `canUseDiskGuard` 追加＋実行ログ＋設定 UI） | **P0** |
| L5 | **劣化兆候アラート** | 「あなたの Mac、先月より遅くなっています」を実測で通知（週平均 loadAvg / Swap の悪化検出） | ○（他社にない「正直な健康監視」） | 低（HealthHistory の週次比較のみ。レポート④の通知化） | P1 |
| L6 | **複数 Mac 対応（同一ライセンスで 3 台＋統合レポート）** | ライセンスは署名検証なので複数台は容易。統合レポートは同期基盤が必要 | ○（ターゲット層は複数 Mac 保有率が高い。月額の「席」概念に直結） | ライセンスのみ: 低 / 統合レポート: **高**（iCloud Drive 同期でサーバーレス実装が現実的） | P1（台数許諾）/ P2（統合レポート） |
| L7 | **AI 相談へのレポート文脈注入** | 2.7 参照。「あなたのデータを知っている AI 相談」 | ○ | 低（プロンプトへの JSON 注入） | P1 |
| L8 | **メールレポート** | 週次レポートのメール配信 | △（通知で代替可能。インフラ・プライバシーコスト大） | 高 | P3（保留） |
| L9 | **年額プラン ¥3,980** | 後述 4 章 | ○（買い切りとの橋渡し） | 低（Stripe Payment Link 追加） | **P0** |

**P0 の合わせ技が重要**: L1（履歴の壁）+ L2（レポート）+ L3（大物拡充）+ L4（DiskGuard）は互いに強化し合う。レポートが「DerivedData +8.4GB」と指摘 → ワンクリック回収（Pro）→ 実績が累計に積まれる → 履歴が育つ → 解約すると失う、というループが閉じる。

## 4. 価格戦略: 月額主軸と買い切りの棲み分け

### 4.1 買い切り ¥4,980 は「廃止せず、意味を変える」

**廃止しない理由:**
- ターゲット（サブスク疲れした技術系ビジネスマン）には買い切り選好層が確実にいる。選択肢を消すと競合（買い切り型ユーティリティ）に流れる。
- 「買い切りも正直に用意している」こと自体がブランド資産。突然の廃止は不信を生む。

**ただし据え置きのまま「意味」を再定義する:**

| | Pro 月額 ¥480 | Pro 買い切り ¥4,980 |
|---|---|---|
| 位置づけ | **育ち続けるプラン** | **現時点の Pro 機能の永久ライセンス** |
| 含まれるもの | Pro 全機能＋**今後追加される検出ルール・レポート項目・新機能をすべて含む** | 購入時点のメジャーバージョン（v1.x）の Pro 機能＋不具合修正 |
| 含まれないもの | — | 次期メジャー（v2）の新機能（優待アップグレード価格を提供） |

つまり**買い切り = 機能のスナップショット、月額 = 継続開発への参加**という業界標準（Sketch 型）の整理。これなら「買い切りを買った人を裏切らない」正直さを保ちつつ、月額に構造的優位を持たせられる。

### 4.2 価格アンカーの再設計: 年額の導入（推奨）

現状は ¥480×10.4 ヶ月 = ¥4,980 で買い切りに並ぶため、11 ヶ月以上使うなら誰でも買い切りが合理的 → 月額が育たない。ここに**年額 ¥3,980（¥332/月相当、月額比 31% off）**を挟む:

- 提示順: **月額 ¥480 / 年額 ¥3,980（おすすめ）/ 買い切り ¥4,980**
- 年額は買い切りより ¥1,000 安く、かつ「新機能全部入り」。合理的な人ほど年額を選ぶ構造になる。
- 買い切りは「サブスク絶対嫌」層の受け皿として温存。買い切り単価が LTV 上限（約 10 ヶ月分）を規定してしまう問題も、年額比では 15 ヶ月分となり緩和される。
- 実装は Stripe Payment Link を 1 本追加し、`LicenseTier` に `proAnnual`（更新日付き）を足すだけ。

### 4.3 買い切り値上げの検討（任意・フェーズ 2）

レポート機能リリース後、Pro の価値が実際に上がった時点で買い切りを ¥5,980〜¥6,980 に改定する選択肢がある。既購入者は影響なし、改定前に「値上げ予告」で駆け込みを取れる。**値上げは機能追加の実績とセットで行えば誇大にならない。**

### 4.4 転換ファネルの設計

- **Pro トライアル 14 日**（全機能）を導入。ただし**トライアル終了時に履歴を消さない** — Free に戻ると「見えなくなる」だけにする（データはローカルに残る）。「あなたの 14 日分のトレンドがここにあります。見るには Pro」は最も正直で強い転換圧力。
- 転換トリガーの計測: レポート通知タップ率 → レポート閲覧 → ロック要素タップ → 購入画面、のファネルをローカルイベントで記録（外部送信しない。将来の意思決定用にオプトインで集計）。

## 5. 実装ロードマップ

### フェーズ 1 — MVP（目標: 2〜3 週間）

**M1. スキャン結果スナップショットの永続化**（レポートの前提。最初にやる）

## 6. 「正直さ」ブランドを損なわないためのガードレール

1. **Free の実用性は現状維持が下限。** メモリ最適化・診断・スキャン・AI 相談は今後も無制限。壁を作るのは「新たに生まれる価値（履歴・レポート・自動化）」の側だけで、既存の無料価値を後から取り上げない。
2. **予測・推定には必ず条件を付ける。** 「このペースが続けば」「あなたの実測では」。断定形の予測（「3 週間後に一杯になります」）は禁止し、レポート文言はテンプレートでレビュー・統制する。
3. **「問題なし」を言えるレポートにする。** やることがない週は「今週は健康です」と報告し、無理に不安を煽って削除させない。恐怖マーケティングをしないことが、クリーナー系との最大の差別化を毎週更新する。
4. **解放量は実測のみ表示。** レポート・通知・購入訴求のどこにも「最大〜GB 解放可能」という推定値を使わない。使うのは「先週実際に解放した量」。
5. **ロック UI は「隠す」のではなく「あることを示す」。** Free ユーザーにもグラフの輪郭やトレンドの存在は見せ、値と解釈だけをロックする。「何が有料か」が常に透明であること自体が正直さの表現。
6. **データはローカルに留める。** レポートも履歴も端末内で完結（メール配信を除く）。オンデバイス主義はプライバシー面の正直さとして LP で明言する価値がある。
7. **通知は週 1 が原則。** 通知の頻度と内容を設定で完全にコントロール可能にする。継続価値のための通知が通知疲れによる解約・アンインストールを招いては本末転倒。
